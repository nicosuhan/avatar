\chapter{Methodology}

\section{Baseline}

\quad The reference baseline for this project is the trained-from-scratch
Transformer used in the AVATAR paper \cite{ahmad2023avatar}. It is kept as a
comparison point because it uses the same benchmark and the same translation
directions, but it does not benefit from code pretraining or instruction
tuning. The model is a standard encoder-decoder Transformer
\cite{vaswani2017transformer}: the encoder reads the token-spaced source
program, the decoder generates the token-spaced target program, and
self-attention allows both sides to model dependencies between distant tokens.

The local baseline script follows this architecture with six encoder layers,
six decoder layers, shared token embeddings, AdamW optimization, label
smoothing, and beam search. Its role in the thesis is not to be the main
system, but to show the starting point provided by a supervised neural
translator trained only on AVATAR. This makes the later pretrained solutions
easier to interpret: any improvement can be related to the effect of
pretraining, prompting, or parameter-efficient fine-tuning rather than only to
the dataset itself.

\section{Zero-Shot Solution with Qwen2.5-Coder}
\label{sec:zero-shot-qwen}

\quad The first implemented solution is a zero-shot translator based on
Qwen2.5-Coder-7B-Instruct. Qwen2.5-Coder is a family of code-oriented large
language models trained for code generation, reasoning, and repair
\cite{hui2024qwen25coder}. The selected checkpoint has 7.61 billion
parameters \cite{qwen2024qwen25coder7b}. This setting is called zero-shot
because the model is not fine-tuned on AVATAR and the prompt contains no
worked examples from the benchmark.

For each test program, the script sends the token-spaced source program to the
chat model with a direction-specific instruction. The prompt asks the model to
reconstruct the source program internally and return complete normal source
code in the target language. Additional rules discourage source-language
leakage: Java-to-Python prompts reject constructs such as \texttt{Scanner} and
\texttt{System.out.println}, while Python-to-Java prompts reject constructs
such as \texttt{def}, \texttt{range}, and Python exponentiation syntax.
Decoding is deterministic, with sampling disabled, so the same input and
checkpoint produce the same translation.

The zero-shot pipeline deliberately separates generation from AVATAR
formatting. Early tests showed that directly asking Qwen to generate the
dataset token stream was less reliable than asking for normal code. Therefore,
the final script first generates ordinary Python or Java code, then applies a
deterministic tokenizer that converts the output back into AVATAR's
token-spaced representation. This design tests the ability of a modern
instruction-tuned model to perform code translation without supervised
benchmark training. The full prompt templates are included in Appendix
\ref{appendix:zero-shot-prompts}.

\section{Encoder-Only Solution with UniXcoder}

\quad The second solution studies an encoder-only pretrained model. It uses
\texttt{microsoft/}\allowbreak\texttt{unixcoder-base} \cite{guo2022unixcoder}
as the source-side encoder and adds a newly trained Transformer decoder for
generation. This design answers a specific question: how much does a pretrained
code representation help if only the encoder is reused and the target-language
generator is learned from AVATAR?

The UniXcoder encoder receives the token-spaced source program and produces
contextual source states. The added decoder is a four-layer Transformer decoder
with causal self-attention, cross-attention over the encoder states, learned
target positions, and an output projection tied to the encoder token
embeddings. During training, the target program is shifted in the usual
teacher-forcing setup: the decoder sees the previous target tokens and learns
to predict the next one. During generation, it performs greedy autoregressive
decoding until the end token or the maximum target length is reached.

This approach is intentionally asymmetric. The source side starts from a
pretrained code model, while the target side has to learn syntax, formatting,
and translation behavior from the parallel AVATAR examples. It is therefore a
useful middle point between the scratch Transformer baseline and fully
pretrained encoder-decoder models. If the encoder-only model performs well, the
source representation alone is carrying much of the translation signal; if it
performs weakly on execution metrics, the missing pretrained decoder is a likely
limitation.

\section{Decoder-Only Solution with Qwen2.5-Coder}

\quad The third solution fine-tunes a decoder-only instruction model:
Qwen2.5-Coder-0.5B-\allowbreak Instruct \cite{qwen2024qwen25coder05b}. Unlike the
zero-shot system, this model is trained on AVATAR examples. Unlike the
encoder-only system, it has no separate encoder; translation is represented as
conditional language modeling over a prompt and a target token stream.

Each training example is converted into a chat-style prompt that asks for a
translation from token-spaced source code into token-spaced target code. The
prompt tokens and the target tokens are concatenated into one causal sequence.
The loss is applied only to the target part, so the model learns to continue
the prompt with the correct translation rather than to reproduce the prompt
itself.

The implementation uses a QLoRA-style setup \cite{dettmers2023qlora}: the base
model is loaded in 4-bit precision, and only small LoRA adapters
\cite{hu2022lora} are trained on selected attention projection matrices. This
makes the decoder-only experiment practical on Kaggle hardware while still
testing whether an instruction-tuned causal code model can adapt to AVATAR's
exact token representation. The method also exposes a tradeoff: the model
already has strong code knowledge, but the causal prompt format must fit both
the source program and the target program inside one context window.

\section{Seq2Seq Solution with CodeT5+}

\quad The fourth solution is a pretrained encoder-decoder translator based on
CodeT5+ 220M \cite{wang2023codet5plus}. This model is closest to the classical
neural machine translation formulation used by the baseline, because one side
encodes the source program and the other side decodes the target program. The
main difference is that both sides start from code pretraining rather than from
random initialization.

The selected checkpoint, \texttt{Salesforce/codet5p-220m}, is loaded through
the Hugging Face \texttt{AutoModelForSeq2SeqLM} interface
\cite{wolf2020transformers}. The encoder receives the AVATAR token-spaced
source program, and the decoder is trained to generate the token-spaced target
program directly.

The method uses the standard conditional generation objective for
encoder-decoder models. During training, the encoder input is the source token
sequence and the decoder target is the shifted target token sequence. During
generation, the encoder states remain fixed while the decoder produces the
translation autoregressively. This keeps the translation task explicit: the
source program and target program are represented as two separate sequences,
instead of being packed into one causal prompt.
